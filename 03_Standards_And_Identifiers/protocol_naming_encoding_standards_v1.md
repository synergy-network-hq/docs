It defines how protocol entities — addresses, monetary amounts, cryptographic algorithm identifiers, raw byte data, timestamps, and environment names — must be represented in all contexts: source code, documentation, APIs, configuration files, user interfaces, and inter-system communication. Consistent representation of protocol entities is a prerequisite for interoperability, auditability, and correctness. Implementations that deviate from these conventions introduce ambiguity, silent compatibility failures, and documentation drift. All Synergy Network software, SDKs, wallets, explorers, and documentation must conform to the standards defined herein. 0.2 How to Use This Document Each Part of this document is self-contained and addresses a specific category of protocol entity. Cross-references between Parts are made explicit where relevant. When a Part defines both a "documentation form" and a "code form" for the same entity, both are normative — the correct form depends on the context of use as defined in that Part. Appendix A provides a single-page quick-reference cheat sheet summarising the most commonly needed conventions across all Parts. 0.3 Constituent Standards This consolidated document incorporates and supersedes the following individual standards: SNTS-01 Synergy Network Address Formatting Specification (Revised 12/06/2025) SNTS-04 SNRG Denomination &amp; Precision Specification v1.2 (December 2025) SNTS-06 Post-Quantum Cryptography Naming &amp; Formatting Standards SNTS-07 Binary &amp; Hexadecimal Encoding Conventions SNTS-08 Timestamp Representation Standard SNTS-09 Network Environment Identifier Conventions 0.4 Companion Standards The following standards are referenced by this document but are maintained separately: SNTS-03 AIVM Execution Determinism Standard SNTS-05 Synergy Cross-Chain Protocol (SXCP) Specification 0.5 Revision History Version Date Summary 1.0 March 2026 Initial consolidated release combining SNTS-01, SNTS-04, SNTS-06, SNTS-07, SNTS-08, and SNTS-09. Part I — Address Formatting (SNTS-01) 1.1 Overview Every protocol entity on the Synergy Network that requires a persistent, verifiable on-chain identity is assigned a typed address. All addresses use Bech32m encoding and are derived from FN-DSA-1024 (formerly Falcon-1024) quantum-safe keypairs. The address type is encoded in a human-readable prefix that precedes the Bech32m data, making the address type immediately legible to both humans and software without requiring additional metadata. Bech32m encoding provides case-insensitive, error-detecting representations that eliminate the class of errors introduced by mixed-case checksum schemes. The encoding uses only alphanumeric characters from a restricted alphabet, making addresses safe in all contexts including URLs, QR codes, and printed material. Unauthorized introduction of new address prefixes or repurposing of reserved prefixes, requires formal DAO approval. All changes to the prefix registry must be accompanied by an SNTS amendment. 1.2 Address Type Registry The table below is the complete, authoritative prefix registry. Part II — Denomination &amp; Precision (SNTS-04) 2.1 Purpose and Scope This Part defines the official denomination hierarchy, precision model, and cross-chain decimal conversion rules for the Synergy Token (SNRG), the native asset of the Synergy Network. These rules govern the Synergy Network L1, AIVM smart contracts, all wallets, explorers, SDKs, all SXCP multi-chain integrations, and off-chain systems such as exchanges and price oracles. This standard adopts the Wei-class naming system and establishes nano- wei ( nWei ) as the smallest indivisible monetary unit. 2.2 Monetary Parameters Parameter Value Native Asset SNRG ( Synergy Token ) Base Denomination SNRG Precision 9 decimal places Smallest Unit nWei (nano- wei ) Maximum Total Supply 12,000,000,000 SNRG (12 billion SNRG) Maximum Total Supply ( nWei ) 12,000,000,000,000,000,000 nWei (12 × 10¹⁸ nWei ) All protocol-level amounts must be stored as integer nWei values. Floating-point arithmetic is strictly prohibited in consensus systems. 2.3 Denomination Hierarchy Synergy Network uses a clean, deterministic Wei-class hierarchy. "Wei" is pronounced "way" (named after Wei Dai): mWei = milli-way, µWei = micro-way, nWei = nano-way. Deterministic division with explicit rounding (floor on precision reduction). No floating-point operations permitted. All user-facing values must be converted from integer nWei at the display layer. 2.4.3 Wallet &amp; Explorer Display Default display unit: SNRG. Advanced mode units: mWei , µWei, nWei . Part III — Post-Quantum Cryptography Naming Standards (SNTS-06) 3.1 Overview and Scope This Part defines the canonical naming and formatting conventions for all NIST-standardized post-quantum cryptographic algorithms used within the Synergy Network and its associated software. These conventions ensure that algorithm names are written consistently across documentation, code comments, source files, and filenames — eliminating ambiguity between formal NIST names and legacy algorithm names. The Synergy Network implements a six-family post-quantum algorithm suite through the Aegis PQC Core subsystem. Two of the six families — SLH-DSA and CMCE-KEM — are included in the Aegis specification as reserve algorithms but are not used in active protocol operations due to operational constraints (signature size and key size respectively). The naming conventions in this Part apply to all six families regardless of their active/reserve status. 3.2 General Rule: Documentation Form vs. Code Form Every algorithm has two canonical forms: Documentation form: used in all written documentation, inline code comments, API reference text, and user-facing strings. This is the official NIST FIPS name with standard hyphenation and capitalisation . Code form: used in all source file names, variable names, function names, struct/type names, and other code identifiers. This is a lowercase, hyphen-free version derived directly from the documentation form. Legacy algorithm names (CRYSTALS-Kyber, CRYSTALS- Dilithium , Falcon, SPHINCS+, HQC, Classic- McEliece ) must not be used in any new documentation or code. Existing references to legacy names should be updated on next revision. 3.3 ML-KEM (Key Encapsulation — formerly CRYSTALS-Kyber) ML-KEM is the primary key encapsulation mechanism in the Synergy Network. It is used for secure channel establishment between validators, session key derivation, and entropy seed generation. Part IV — Binary &amp; Hexadecimal Encoding Conventions (SNTS-07) 4.1 Scope This Part governs how raw binary data — cryptographic hashes, public keys, signatures, and other byte arrays — is serialized and displayed across all Synergy Network contexts: JSON APIs, documentation, logs, on-chain storage, and inter-process communication. Address encoding is governed separately by Part I (Bech32m); this Part covers all other binary objects. 4.2 General Hex Encoding Rules All binary data, when represented as text, uses lowercase hexadecimal encoding with no prefix. The following rules are normative: Characters must be from the set [0-9a-f] only — uppercase hex characters [A-F] are not permitted in any Synergy-native encoding context. No prefix of any kind is prepended to hex strings. The 0x prefix convention is not used anywhere in the Synergy protocol. Hex strings are always full-length and zero-padded to their expected byte-count × 2 characters. Leading bytes of zero value are never omitted. The character length of a serialized hex string is always exactly: byte_length × 2. The 0x prefix, mixed-case checksummed hex (EIP-55 style), Base58, Base58Check, and Base64 are not used in any Synergy-native encoding context. These formats may appear only at external-chain compatibility boundaries and must be explicitly marked as compatibility translations. 4.3 Typed Hash Identifiers Protocol-level hashes that appear in user-facing contexts (explorers, APIs, log output) carry a typed prefix that identifies the entity type the hash represents. The prefix convention established by the protocol is syn{type}-{hex}. Prefixed and raw hex forms each have their appropriate context: Prefixed form (e.g. syntxn-a0d53ef9...): used in user-facing contexts — explorer UIs, API responses, documentation examples, and log messages where the entity type must be unambiguous. Raw hex form (e.g. a0d53ef9...): used in machine-to-machine contexts such as internal node state, binary serialization, and inter-process communication where the type is unambiguous from structure. Identifier Prefix Hex Length Represents Transaction hash syntxn - 64 chars Hash digest of the serialized transaction. Block hash synblk - 64 chars Hash digest of the block header. Merkle root synmkl - 64 chars Merkle root of the transactions contained in a block. Epoch randomness beacon synepoch - 128 chars SHA3-512 entropy beacon output at an epoch boundary. Quorum certificate hash synqc - 64 chars Hash of a finalized Quorum Certificate. 64 hex characters = 32 bytes, consistent with SHA3-256 output. 128 hex characters = 64 bytes, consistent with SHA3-512 output used in the PoSy entropy beacon. 4.4 Public Key Serialization Public keys are serialized as lowercase hex when represented as text. No prefix or wrapper is applied. The expected character count is determined by the algorithm variant and is fixed — implementations must validate length on deserialization. Part V — Timestamp Representation (SNTS-08) 5.1 Wire Format All timestamps in the Synergy protocol are Unix epoch timestamps expressed as an unsigned 64-bit integer (u64) counting whole seconds since 1970-01-01T00:00:00Z. Millisecond, microsecond, and nanosecond precision is not used at the protocol layer. Fractional seconds are never transmitted. // Rust wire type timestamp: u64 // whole seconds since Unix epoch // JSON API type "timestamp": 1741561800 // integer, never a float or string 5.2 Precision Rationale Second-level precision is sufficient for all Synergy protocol operations. The minimum meaningful time unit in the PoSy consensus protocol is the block interval (approximately 2 seconds). The replay-attack prevention window for inter-cluster messages is 5 seconds. Sub-second precision would add wire overhead with no protocol benefit and would introduce unnecessary dependency on clock synchronization quality across heterogeneous validator hardware. Millisecond timestamps (the JavaScript Date.now () convention and the Ethereum block timestamp convention in some tooling) are not used. Implementations that receive a millisecond-scale integer should reject it as malformed rather than silently accepting a timestamp 1,000× larger than expected. 5.3 Display Format in Documentation and UI When timestamps are shown to humans — in documentation examples, block explorer displays, or log output — the canonical display format is ISO 8601 UTC with a Z suffix and no fractional seconds: 2026-03-10T14:30:00Z // NOT: 2026-03-10T14:30:00.000Z (no fractional seconds) // NOT: 2026-03-10T09:30:00-05:00 (no local timezone offsets) // NOT: March 10, 2026 (no informal date strings) Local time representations must not be used in any protocol-facing display. All times are UTC, always. 5.4 Timestamp Field Naming Conventions Timestamp fields in all protocol data structures and API responses follow a consistent naming scheme: Context Convention Examples Current/single timestamp timestamp timestamp, block_timestamp Event time {event}_at created_at , finalized_at , registered_at JSON field names snake_case " created _at ": 1741561800 Rust field names snake_case pub created_at : u64 Fields must never use millisecond-suggesting suffixes such as timestampMs , ts_ms , or timestamp_millis . If a field name contains " ms " it implies the value is in milliseconds — which is not a valid Synergy timestamp format.

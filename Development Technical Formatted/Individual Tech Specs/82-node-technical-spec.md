Abstract This document defines the Synergy Network node operating model in a way that is deliberately aligned to the current architecture corpus rather than to legacy draft assumptions. It establishes a canonical five-class, nineteen-role node taxonomy; defines the configuration architecture expected to govern those roles; explains how each node participates in the identity, routing, and execution planes; and specifies the Synergy Node Control Panel as the no-code operator surface through which Jarvis provisions, validates, and maintains compliant nodes. B. Executive Summary Synergy Network is not a one-daemon system with a flat validator list and a handful of convenience sidecars. It is a layered protocol stack in which different node roles are intentionally separated so that authority, observability, execution, interoperability, and operator access do not collapse into one another. The consequence is straightforward: operators should not think in terms of “running the blockchain” as a monolith, but in terms of running clearly bounded protocol services that each inherit only the authority they actually need. The core design decision throughout this specification is that nodes may provide availability, verification, storage, routing, analytics, or emergency participation, but those capabilities do not automatically imply sovereign authority. Relayers do not become bridges. Wallet-facing nodes do not become trust anchors. Governance-security nodes do not become override shells. Service nodes do not become validators. The node model mirrors the whitepaper’s broader discipline: identity is not execution; routing is not authority; execution is not identity. C. Purpose of the Node Technical Specification The purpose of this specification is to answer a very practical question with architectural precision: what exactly does each Synergy node role do, what authority does it have, what software surfaces does it run, what configuration material does it require, and how is that role made operable for a real-world node operator without collapsing the protocol’s security model? That is the problem space this document solves. 2. Relationship to the Whitepaper and Subsystem Specifications This document is subordinate to the architectural invariants established by the Synergy whitepaper corpus. It does not redefine PoSy, Aegis, SXCP, SCETP, SynQ, UMA, the wallet planes, or governance authority. Instead, it translates those architectural commitments into node-role language: executable services, configuration boundaries, deployment expectations, observability surfaces, and operator workflows. 3. Node Taxonomy, Authority Planes, and Class Boundaries Synergy’s nineteen node roles are organized into five classes. That taxonomy is not a marketing taxonomy. It is a risk-separation taxonomy. Class boundaries exist because authority domains should not casually overlap. Class Purpose Authority Character Class I — Consensus Nodes Produce, validate, and audit PoSy consensus artifacts Highest protocol-criticality; finality-adjacent Class II — Interoperability Nodes Observe, relay, verify, and map external-domain facts Bridgeless, attestation-based, non-custodial Class III — Execution, Data, and Cryptography Nodes Execute deterministic workloads, maintain DA, and run high-assurance crypto services Authority bounded by Aegis / execution policy Class IV — Governance and Security Nodes Verify and execute bounded governance and emergency actions Constrained collective authority only Class V — Service and Access Nodes Expose user-, wallet-, and operator-facing network services Service-plane only; non-sovereign Across every class, the same plane discipline applies. Nodes may touch identity state, route intent, or authorize execution only to the extent their role explicitly requires. A service node that routes requests must not silently gain execution power. A governance node that helps contain emergencies must not become a general override channel. A relayer that improves availability must never become a bridge. 4. Global Node Lifecycle and Deployment States Every Synergy node role moves through the same lifecycle: enrollment, profile selection, key material binding, policy validation, sync / bootstrap, active participation, rotation or maintenance, deactivation, and audit-preserving retirement. The role-specific behavior differs, but the lifecycle shape remains consistent so that the Node Control Panel, governance, and security tooling can reason about any node in a common way. Lifecycle State Meaning Unenrolled Software installed but not identity-bound and not authorized to participate Provisioned Profile selected, base config written, secrets enrollment pending or complete Syncing Node is catching up, validating dependencies, and not yet participating in its live role Active Node is in-role and may produce or serve artifacts allowed by its class Quarantined Role participation suspended due to policy failure, security event, or operator action Retired Node removed from active service with logs, receipts, and deactivation artifacts preserved 5. Canonical Configuration Architecture A consistent config architecture is the difference between an operable network and a mess of snowflake deployments. This specification therefore defines a canonical hierarchy built around a shared base file plus role overlays. The goal is straightforward: every node should read like an instance of the same platform, with only bounded differences by role. Security, Identity, and Aegis Requirements for All Nodes No Synergy node is permitted to treat raw key possession as a substitute for protocol-recognized authority. Identity binding is UMA-aware where applicable, cryptographic roles are Aegis-governed, and key lifecycle controls are generation-scoped. At the operator level, that translates into a short but brutal rule: if your node package or tooling tries to make “having the key” equivalent to “being allowed to act,” then the package is wrong. All roles that sign, attest, authorize, or participate in threshold workflows must use role-bound cryptographic material. HSM-backed or device-isolated key paths are strongly recommended for governance, treasury, and emergency roles and are generally expected for anything with high blast radius. Secret material must not be stored in the same plaintext context as exported profile files, screenshots, or generalized system backups. The Node Control Panel may automate enrollment, but it must never erase the underlying boundary between operator convenience and cryptographic authority. 7. Observability, Logging, Auditing, and Failure Containment Synergy’s failure philosophy is simple: do nothing unsafe, loudly. Nodes therefore emit structured logs, signed receipts where appropriate, health metrics, and role-aware audit artifacts. Failure should collapse capability, not create a hidden fallback path. A relayer that cannot establish external finality context should stop. A validator with a corrupted state DB should quarantine itself. A gateway that loses upstream trust should degrade into refusal rather than silently serving stale or ambiguous data. The Control Panel, governance auditors, and operators all rely on observability artifacts that are durable, attributable, and parseable. Minimal expectations include structured JSON logs, health endpoints, alertable service states, rotation and deactivation receipts, immutable governance / treasury execution logs where applicable, and signed update provenance for anything that can change the role behavior of a live node. 9. Consensus Node Class Overview Consensus Nodes secure the chain through PoSy. They are closest to finality and therefore carry the strictest uptime, state-integrity, and cryptographic-verification expectations. Not every consensus-adjacent node is identical, however: some produce and vote, some archive, and some audit. This distinction matters because separating live finality authority from history retention and audit visibility is one of the simplest ways to reduce blast radius. 10. Product Definition and Operator Positioning The Synergy Node Control Panel is the operator-facing control plane for Synergy nodes. It is not a cosmetic dashboard layered on top of shell scripts. It is the official no-code node operations surface through which operators provision node roles, bind identity safely, apply policy-constrained role profiles, monitor health, rotate software, export diagnostic bundles, and coordinate security-sensitive actions without dropping into terminal-first workflows. Jarvis is the orchestration assistant inside that control plane. Jarvis exists to remove operational friction, not to remove operator responsibility. It asks for the minimum correct inputs, renders architecture-safe choices as guided flows, writes bounded configs, validates prerequisites, and refuses unsafe combinations. A user should not have to write a line of code to operate a compliant node. That is a product requirement, not a nice-to-have. 34. Jarvis-Guided No-Code Provisioning Workflow Phase Jarvis Responsibility Host qualification Checks CPU, RAM, disk, network reachability, clock sync, virtualization posture, and environment profile fit Role selection Presents only the node profiles appropriate to the environment and operator intent Identity and security binding Guides UMA / role binding, Aegis enrollment, HSM or device-isolated key paths, and secret handling Config generation Produces node.toml + role overlays + secure references rather than exposing raw schema first Bootstrap Downloads signed artifacts, validates checksums, starts initial sync, and tracks readiness gates Operations Monitors health, recommends updates, surfaces alerts, and can enter guided maintenance or quarantine flows Jarvis should operate like an opinionated SRE assistant that understands protocol constraints. It must never offer “advanced freedom” that accidentally bypasses authority boundaries. The UX goal is not endless customization; it is safe operability with intelligent defaults and bounded expert override. 35. Control Plane Architecture and Trust Boundaries The Control Panel itself must respect the same plane separation visible in the protocol. A UI that can silently mutate identity state, rewrite execution policy, or export signing material without traceability is not a node tool; it is an attack surface. Accordingly, the Control Panel is composed of a local app shell, a role-profile engine, a secure secret broker, an update verifier, a metrics/log collector, and a Jarvis reasoning layer that consumes state but does not itself hold sovereign authority. For example, a Devnet validator template can expose experimental logging or sync instrumentation that Mainnet would not. A Treasury Controller template must force HSM-backed signing paths, threshold policy configuration, and stricter update controls than a read-only explorer. Templates are therefore not merely convenience presets; they are productized policy boundaries. # control-panel profile bundle (conceptual) profile_id: validator-devnet role: validator environment: devnet requires: - aegis - telemetry - signed_updates - quorum_health_checks blocks: - plaintext_key_export - unsafe_remote_admin recommends: - hsm_or_device_isolated_keys - archive_peer_pairing 37. Secret Handling, Recovery, and Safe Automation The Control Panel must treat secret handling the same way the wallet treats authority: convenience never overrides structural security. Secrets should be enrollable through guided hardware or device-bound paths, referenceable through secure handles, and exportable only through explicit, audited, policy-approved flows. Jarvis can automate rotations, renew certificates, refresh overlays, or step operators through recovery and rebind workflows, but it must not normalize seed-like plaintext handling, “copy this key into a box” UX, or unconstrained remote admin shells. For operators using the Control Panel to run governance or treasury roles, safe automation means threshold-aware prompting, quorum-health validation, timelock visibility, and pre-execution policy simulation. For consensus or interoperability roles, it means drift detection, environment-correct port and peer overlays, and immutable release provenance. In both cases, the software should do the boring work automatically and refuse the dangerous work automatically. 38. Updates, Monitoring, Alerts, and Supportability A credible node product needs a real lifecycle. The Control Panel should therefore implement signed update channels, staged rollout support, health scoring by role, guided log-bundle export, machine-readable alerts, and operator-facing incident narratives that explain not just that something is wrong, but what boundary failed and which safe next actions are available. Jarvis should speak operator language, not dump raw stack traces unless the operator explicitly requests them. Hardware, Storage, and Capacity Guidance Hardware requirements vary by role, but the governing principle is not raw horsepower; it is role fit. Consensus and verification roles need reliable CPU, storage IOPS, network stability, and high-integrity timekeeping. Archive and DA roles need storage depth and retention strategy. Service roles need edge resilience and rate-limit capacity. Governance and treasury roles need security posture first and performance second. Security Hardening Baseline • Run all nodes with least-privilege service accounts and explicit file ownership boundaries. • Expose only required ports; keep metrics, KMS, and administrative surfaces bound to localhost or private networks unless a role explicitly requires otherwise. • Require signed updates and verify release provenance before package install or in-place upgrade. • Use environment overlays; never point a mainnet role at devnet endpoints or vice versa. • Prefer HSM- or device-isolated signing for governance, treasury, emergency, and high-blast-radius interoperability roles. • Emit structured logs and retain signed or immutable audit receipts for privileged actions. • Quarantine on integrity failure, role mismatch, or cryptographic policy drift instead of continuing optimistically. 43. Glossary of Node-Specific Terms The following terms have specific, bounded meanings within the Synergy node operating model. Where a term appears in a role specification or configuration schema, its definition here is canonical. Aegis The post-quantum cryptographic core of Synergy Network. Aegis governs all signing, key encapsulation, lifecycle management, and threshold authorization. It is a jurisdictional plane, not a library — no node may redefine its semantics locally. Aegis Cryptography Node Class III node role responsible for providing Aegis verification services, KMS bridging, and key lifecycle management to other nodes on the network. Archive Validator Node Class I node role that maintains a complete historical state and block corpus, enabling proof reconstruction and long-horizon auditability without participating in active consensus. Audit Validator Node Class I node role that runs independent verification of consensus artifacts and quorum certificates to detect divergence, equivocation, and implementation defects. Authority Plane A functional domain within the Synergy protocol that holds a distinct type of protocol authority: identity, routing, or execution. Nodes are bounded to the plane(s) their role explicitly requires. Bootnode A well-known node used exclusively for peer discovery during network bootstrap. Bootnodes do not participate in consensus. Class (Node Class) A grouping of node roles sharing a common authority domain and operational posture. The five classes are: Consensus, Interoperability, Execution/Data/Cryptography, Governance/Security, and Service/Access. Committee Node Class I node role that executes committee-assigned PoSy duties within a given epoch, including cooperative voting and cluster coordination. Data-Availability Node Class III node role responsible for storing, serving, and verifying data-availability material needed to keep execution auditable at scale. Epoch A time-bounded operational period in PoSy, approximately 1,000 blocks. Epoch boundaries trigger committee rotation, key lifecycle events, and governance lockup calculations. Environment Overlay A configuration layer that applies environment-specific values (endpoints, ports, flags) on top of a shared base configuration. Valid environments: devnet, testnet, mainnet-beta, mainnet. Governance Auditor Node Class IV node role that verifies proposal typing, execution scope, Synergy Score weighting, and governance artifact integrity. Indexer and Explorer Node Class V node role that indexes blocks, events, and state changes for query, search, and operator observability. Non-authoritative. Jarvis The orchestration assistant embedded in the Synergy Node Control Panel. Jarvis provides guided, no-code node provisioning and monitoring without removing operator responsibility. KMS Key Management Service. The Aegis KMS bridge provides hardware-backed or device-isolated key operations for roles that require signing authority. Node Role A specific, bounded operational identity within a node class. Each role has a defined service surface, authority scope, configuration profile, and failure posture. Observer / Light Node Class V node role that verifies headers and lightweight proofs for wallets, agents, and low-resource devices without full-state participation. Oracle Node Class II node role that provides authenticated external data attestations for narrowly scoped use cases where protocol policy explicitly allows extra-chain reference data. PoSy (Proof-of-Synergy) The Synergy consensus protocol. PoSy uses cooperative validator clustering, epoch-scoped committee rotation, Synergy Score weighting, and Aegis-backed quorum certificates to achieve deterministic finality under partial synchrony. Quarantine The safe failure mode for a Synergy node: halt execution and emit a structured alert rather than continue operating with violated invariants. Quarantine is always preferred over optimistic continuation. Relayer Node Class II node role that propagates cross-chain observations and packages attestation material for SXCP without acquiring custody or delegated execution authority. Role Overlay A TOML configuration file (roles/&lt;role_name&gt;.toml) that applies role-specific settings on top of the shared node.toml base configuration. RPC Gateway Node Class V node role that exposes rate-limited public RPC and WebSocket access without inheriting consensus or governance authority. SCETP Synergy Same-Chain External Transfer Protocol. Governs authority ingress for external operations executed on Synergy itself. Security Council Node Class IV node role that participates in time-bounded, governance-constrained emergency actions such as scoped pauses and cryptographic deactivation. Service Surface The set of protocol interfaces a node role exposes (e.g., p2p, rpc, ws, aegis-verify). A node may not expose surfaces outside its defined service surface. Synergy Node Control Panel The official no-code operator interface for provisioning, monitoring, and managing Synergy nodes. Backed by the Jarvis orchestration assistant. Synergy Score A composite, epoch-scoped authority signal derived from economic commitment, cooperative participation, and longitudinal behavior. Used for validator weighting, governance influence, and cluster formation. SXCP Synergy Cross-Chain Protocol. The interoperability protocol enabling deterministic cross-chain attestation, atomic swap, and threshold coordination without pooled custodial bridges. SynQ Execution Node Class III node role that hosts the SynQ smart contract virtual machine and executes formally verified, deterministic contract logic. Treasury Controller Node Class IV node role that executes approved treasury and compensation-pool actions under bounded governance scope and threshold-controlled authorization. UMA (Universal Meta-Address) A persistent, chain-agnostic identity anchor that survives key rotation events. UMA is the binding primitive for the Synergy identity plane. UMA Coordinator Node Class II node role that coordinates UMA identity bindings, domain mappings, and identity-safe refresh events. Validator Node Class I node role that participates in PoSy propose-vote-commit consensus, validates state transitions, and signs finalized quorum material. Witness Node Class II node role that observes finalized external-chain facts and emits verifiable witness evidence into SXCP pipelines.
